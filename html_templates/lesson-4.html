Important Concepts

Lesson 4.1: Introduction to Networks

The internet is a network. A network is a group of entities that can communicate even though they are not all directly connected. Networks are measured with latency and bandwidth.

  • Latency: Time it takes for a message to get from its source to its destination. It is measured in time (milliseconds).
  • Bandwidth: Amount of information that can be transmitted per unit time (Mbps).
  • Bit: 1 bit = smallest unit of information.
  • Protocol: A set of rules that people agree to that defines how two entities, a client and a server, can talk to one another. Hypertext Transfer Protocol, or HTTP, is the protocol for the client and the server.

Lesson 4.2: Make the Internet Work for You

  • Uniform Resource Locator (URL): A combination of the protocol, the host, and the path. For instance, in 'http://www.udacity.com/cs253', http is the protocol, www.udacity.com is the host, and cs253 is the path.

  • Query Paramters: Seen after the path in the URL. Multiple query parameters can exist after the path separated by '&' characters. The query parameters send additional information to the server.

  • Cache: Something that stores data so you don't have to retrieve it later. Caches are used to make data requests faster.

  • Fragment: Added to the URL after the path. Fragments are separated by '#' symbols. The fragment is not sent to the server, it exists solely in the browser. When there are query parameters, fragments follow query parameters.

  • Port: Necessary to retrieve data from a server.
  • Requests: A request line has 3 main parts: a method, a path, and a version. Popular methods are 'GET' and 'POST'.
  • Server: A computer that interacts with a request we make to the computer.

    Lesson 4.5: Working with App Engine

    GET and POST are the two most popular methods used to talk to servers. Here are the main differences between them.
    GET:

    • Parameters in URL.
    • Used for fetching documents.
    • Maximum URL length.
    • Ok to cache.
    • Shouldn't change the server.

    POST:
    • Parameters in body.
    • Used for updating data.
    • No max length.
    • Not ok to cache.
    • Ok to change the server.

    Lesson 4.6: Input Validation

    Validation means verifying on the server side that what we receive is what we expected to receive.

    • Input Validation:
      1. Verify the user's input.
      2. On error, render form again.
      3. Include error message.

    • String Substitution: Used to replace parts of strings with other text. This can be done in Python with the modulus '%' operator.

    • HTML Escaping: HTML allows you to change certain characters so that when the user types them in, the server shows them. They are:
      • " : &quot;
      • > : &gt;
      • < : &lt;
      • & : &amp;

    • Redirection
      • Can't share success link.
      • Can't reload the page without annoying message
    Validation is important for both site security and user experience.
    • Site Security: It is important to not leave your website open to attack. Handling HTML escaping is critical to this because failing to do so allows users to modify your website!
    • User Experience: The user might not be able to understand why the server is not accepting their POST request, or the server might allow the user to enter false data.
    • Possible Consequences of Failing to Validate User Input:
      • Website vulnerable to hackers and other malcontents.
      • Poor user experience.
    Servers handle requests and deliver responses by:
    1. The user makes a GET request for the form.
    2. Server responds with form data.
    3. The user then makes a POST request to the server with the data.
    4. The server will then validate the request and either accept the data or not. If there is an error, and the server does not validate the request, the server will send a form request back to the user with an error message. The user can then send another POST request back to the server.

    Lesson 4.7: HTML Templates

    Templates generally refer to a library. A template library is a library to build complicated strings, typically HTML.

    • Variable Substitution: {{ variable }}
    • Statement Syntax:
      • {% statement %}
      output
      {% end statement %}
    • For Loop Syntax:
      • {% for statement %}
      body
      {% endfor %}
    • Helpful Hints:
      • Always automically escape variables when possible.
      • Minimize code in templates.
      • Minimize html in code.
    • Templates:
      • Separate different types of code (Ex: HTML vs. Python)
      • Make more readable code.
      • Make more secure websites.
      • HTML that is easier to modify.
    The Importance of Avoiding Repetition:
  • Makes your code less prone to errors.
  • Easier to fix and modify code.
  • Code is consistent so changes are applied universally.
  • Using HTML templates when creating a website is an important tool to use in order to avoid repetition.

    • Lesson 4.8: Databases

    • What is a database?: A program that stores and retrieves data, usually large amounts of structured data. Many servers are used specifically for databases.
    • Why databases?: Many downsides of query data by hand. Databases exists to take large amounts of data and run queries on them in reasonable periods of time.
      • Error-prone.
      • Tedious.
      • Slow.
    • Relational Databases: Typically use Structured Query Language (SQL) to query them. A few of these are PostgreSQL, MySQL, SQLite, Oracle, and others. MySQL is the most popular database by far, and is owned by Oracle, but is still free and open for the most part.
    • SQL:
      • Invented in the 1970s before the internet, the web, or web applications existed.
      • SQL is fairly readable, simple language.
      • Typical SQL query uses 'SELECT', 'FROM', and 'WHERE' statements. 'SELECT' fetches data, 'FROM' identifies the table we are using, and 'WHERE' puts constraints on the data we are fetching.
    • ACID:
      • A - Atomicity: All parts of a transaction succeed or fail together.
      • C - Consistency: The database will always be consistent.
      • I - Isolation: No transaction can interfere with others.
      • D - Durability: Once the transaction is committed, it won't be lost.
    • Google App Engine Datastore:
      • Tables -> Entities (columns are not fixed, all have an ID, parents/ancestors)
